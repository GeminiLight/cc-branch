Workspace Matrix direction

配置页应该先是一张可编辑的工作空间地图。

用户配置的不是 YAML 节点，也不是列表里的 Tab。用户真正关心的是：打开 Warp、VS Code、Cursor 或 Terminal 后，会出现几排 Tab，每排里面并排几个 Pane，每个 Pane 跑什么。

核心原则Canvas 直接把 Tab 渲染成多排，不要求用户先切换。
推荐方向每一排是一个 Tab，每排内部横向排列 Panes。
降低表单感Inspector 只编辑细节，不负责主要结构调度。
映射后端内部仍可保存为 slots/windows，UI 上显示为 Tab/Pane。

为什么比 App Mirror 更适合配置页

配置时用户需要全局调度，不应该每次只能看一个 Tab。Matrix 能同时看到所有 Tab 和 Pane，心智更像一张地图。

操作模型

点空白加 Tab，点行尾加 Pane，拖 Pane 到另一行就是移动，拖 Tab 行到另一行就是合并。选中 Pane 后只编辑这个 Pane 的名字、Agent、命令、目录。

和真实打开的关系

Matrix 是配置视图；点 Preview opened app 后，再切到 App Mirror，展示真实终端 App 打开后的单窗口效果。

后端映射

UI 术语用 Tab / Pane；保存时仍可写回现有 slots / windows schema，不必马上大改数据结构。

A. App Mirror Preview

打开预览最所见即所得

画布直接模拟用户打开后的终端 App：顶部是真 Tab，内部是真 Pane 分割。它非常适合 Preview，但默认配置页如果只展示当前 Tab，会少一点全局感。

为什么推荐

用户看到的就是打开后的样子，不需要理解“列表项对应哪里”。

交互重点

Tab 是顶部标签；Pane 是画布里的终端分屏。合并 Tab = 把它变成当前 Tab 的 Pane。

落地方式

先不做真实拖拽也可以，用卡片按钮模拟：合并、移动、拆分、复制。

我的推荐

默认配置页用 Workspace Matrix：所有 Tab 以多排形式直接显示，每排内部就是 Panes。它更适合“配置”和“调度”。App Mirror 不删除，但降级为 Preview：用户想确认打开后像不像真实终端 App 时再切过去。

配置页首屏

左侧/中央是 Matrix Canvas：Tab 行和 Pane 卡同时可见。不要默认藏在切换里。

主要动作

加 Tab、加 Pane、移动 Pane、合并 Tab 都在 Canvas 上完成。Inspector 只做字段编辑。

预览模式

保留 App Mirror：它负责模拟真实打开后的单窗口，不负责主要结构编辑。
